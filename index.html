为什么MP4文件录像传输中断后无法播放

Posted on 2025-12-07

MP4文件是由一系列Box组成的结构化容器，其中最关键的两类Box是：

  • mdat（媒体数据）：存放真正的H265/H264/AAC原始数据。
  • moov（索引信息）：存放所有Sample的时间戳、偏移、帧类型等元数据。

在录制过程中，设备会不断写入mdat，但moov一般在录像结束时一次性生成。

在设备突然断电时出现的问题的根本原因是：

  1. mdat已写入，但moov还没来得及生成或写完整。
  2. 没有moov，播放器就无法知道mdat中的帧结构与时间轴，就认为“文件损坏“。
  3. 若moov写了一半，也会导致offset错误、sample table不完整，导致无法播放。

因此，中断造成的MP4损坏，本质上就是：索引信息缺失或不完整，而不是数据本身坏了。

如何修复

  1. 可以通过扫描mdat中的NALU，重新构建sample table，补写一个新的moov box。
  2. ffmpeg的 -movflags + faststart 就是提前写moov的一个思路。
  3. 或者采用 **fragmented MP4(fmp4)**，实时写moof/mdat，提高断电恢复能力。

总结

因为 moov 在录制结束后才写入。
断电 ⇒ moov 缺失 ⇒ 索引不存在 ⇒ mdat 无法被解析 ⇒ 文件看似损坏。

SPS PPS VPS的概念

Posted on 2025-12-07

前言

  • SPS(Sequence Parameter Set)：描述一段视频序列的宏观配置
  • PPS(Picture Parameter Set)：描述一张图像/帧的细化配置
  • VPS(Video Parameter Set)：H265才有，用于描述整个视频层的”顶层配置“

SPS有什么

SPS是描述整段视频公用的编码规则：

  • 宽 X 高（Resolution)
  • 颜色格式
  • 最大参考帧数量
  • 解码所需能力级别（Profile/level)
  • 时序信息（fps相关）
  • 图像切分、宏块结构

作用：解码器根据SPS才知道准备多大的缓冲区、如何组织解码结构。

PPS有什么

PPS描述帧级别的参数：

  • 去块滤波参数(deblocking filter)
  • 熵编码类型(CABAC/CAVLC)
  • 参考索引设置
  • 图像切片结构

作用：为每一帧提供细节。

VPS

H265新增的更顶层参数，内容：

  • 多层结构、子层设置
  • 多码流协同参数（适用于HDR、多码率、多视点）
  • 解码能力需求

作用：H265更复杂，需要一个总导演来协调所有层次，所以出现了VPS。

缺失会发生什么

如果在传输过程中缺失SPS/PPS会出现花屏、解码失败、画面无法启动等错误。想是在flv header中其实就有SPS的值出现。

GOP是什么

Posted on 2025-12-07

GOP的全程是Group of Pictures 指编码器生成的图像序列结构，由I帧、P帧、B帧组成。GOP长度代表两帧（I/IDR）之间的帧数。例如GOP=25表示每秒25fps的视频中，每秒由1帧是I帧。I帧是全量帧，P/B是参考帧，它们依赖GOP结构进行还原。

GOP的长度影响编码效率与抗网络抖动能力：

  • GOP越大（I帧更小）-> 压缩率更高、码率更低、但链路一旦丢包，会导致更长时间的马赛克。
  • GOP越小（I帧更多）-> 恢复快，抗抖动强，但是带宽压力更大。

在弱网环境下，播放器或传输侧可能会：

  • 请求立即发送一个关键帧(FIR/PLI)
  • 锁端GOP周期提高恢复速度
  • 丢弃部分P帧，等待下一个I帧来恢复画面

因此GOP的核心作用是：在压缩效率与弱网恢复速度之间做平衡。

H264、H265与YUV420的压缩对比关系

Posted on 2025-12-07

前言

从sensor采集到图片后经过一系列的操作，我们可以得到一张叫做YUV420的原始图片，它是以一种**未经过压缩的像素格式（RAW)**，祝主要特点有：

  • 每个像素都有亮度Y
  • 色度U/V是按2X2分组取样
  • 所有数据量大约1.5X分辨率，也就是在代码中经常看到的 w * y * 3 /2的原因

例如：

1080p = 1920 x 1080 x 1.5 = 3MB 一帧

30fps时 ： 3MB X 30 = 90MB/s

YUV是一个超大的编码数据，如果我们在不考虑编解码的环境下，单单是一个2Mbit的带宽传输一秒的图片都需要

90 * 8 / 2 = 360s的时间才能传完，延迟极其可怕。

H264/H265的压缩思路

压缩的思路是从巨大的Raw数据中找到规律，丢弃不太重要的信息。

压缩方式主要是一下三点：

  1. 帧内压缩（Intra)——像JPEG一样压缩一帧
    找到图像块的冗余，使用变换 + 量化降低数据。
  2. 帧间压缩（Inter)——利用“前后帧差不多”的事实
    画面差异本来就不大，比如监控摄像头：
    • 背景不动
    • 只有前景改变
      所以只要记录运动矢量 + 残差即可。

这就是压缩率飞起的关键。

  1. Entropy编码
    CABAC/CAVLC等熵编码算法，把高频出现的值压得更短。

压缩效率的对比

编码标准 相同画质下比YUV420减少多少？ 相比前代提升？
H264 大约压缩到 1/20 ~ 1/50
H265 大约压缩到1/50 ~ 1/100 比H264再省30% ~ 50%带宽

例如：

1080p@30fps， Raw约90MB/s

  • H264压缩后：1.5 ~ 4.5MB/s(12~36Mbps)
  • H265压缩后：0.9 ~ 3MB/s(7~25Mbps)

FlvHeader详解

Posted on 2025-11-14

导言

在Flv格式中，Header有两个意思。一个是flv文件的Header，另外一个是每个FlvTag的Header。

Flv文件的Header

长度： 9个字节

Flv文件的Header结构如下：

偏移地址 内容 长度 说明
0x0000 46 4C 56 3 Signature(“FLV”)
0x0003 01 1 Version
0x0004 05 1 Flags(音频+视频)
0x0005 00 00 00 09 4 Offset(DataOffset)

再次之后紧跟着一个长度为4个字节的PreviousTagSize0。
PreviousTagSize0的值为0，表示第一个Tag的Size为0。

Flags字段值： 0x01=audio, 0x04=video, 0x05=audio+video

🧠 一句话口诀：F L V 一五九，前导零结尾。

  • “FLV” → 开头魔数
  • “一” → version 1
  • “五” → flags=0x05（音视频）
  • “九” → header 长度 9
  • “前导零” → 第一个 PreviousTagSize0=0

“Tag 来到” → 紧接第一个 Tag（Script、Video 或 Audio）

FlvTag的Header

长度： 11个字节

偏移 内容 长度 说明
0 8 = Audio
9 = Video
18 = ScriptData
0x12 = Metadata		 1 Tag Type
1 DataSize 3 TagData部分长度（不包含Header/PrevSize）
4 Timestamp 3 时间戳（DTS,低24位）单位：ms
7 TimestampExtended 1 时间戳（DTS,高8位）
8 StreamID 3 保留字段，始终位0X0000

一个Tag的实际总长度为 = 11 + DataSize + 4

PreviousTagSizeN

长度为4个字节，用来标记当前一个FlvTag的长度。

码农之殇，程序员的分水岭

Posted on 2025-10-24

前言

在过去的十年间，”35岁被优化”成为悬在每个程序员头顶的达摩克利斯之剑。传统观念认为，在35岁之前必须完成向技术专家或管理者的转型，否则将面临职业生涯的危机。然而，随着AI时代的来临和自动化编程工具的普及，这一命题正在发生根本性的改变——职业分水岭不再遥远，而是每个程序员都必须立即面对的现实挑战。

码农之殇

时代的变迁：从黄金十年到AI冲击

过去的繁荣景象

回顾计算机行业过去的黄金十年，它一直是高考志愿的热门选择，甚至众多非计算机专业的毕业生也纷纷转型投身程序开发领域。市场的旺盛需求创造了一个独特的就业环境：无论是技术高手还是初级菜鸟，都能在这个行业中找到自己的一席之地。

这种繁荣背后的逻辑很简单：大型项目不仅需要懂业务、懂技术、懂设计交互的架构师，同样需要大量”拧螺丝”的执行者。在这个阶段，程序员的主要工作是实现产品经理的需求，在技术负责人的架构规划下完成CRUD功能和业务逻辑开发。一个在今天看来微不足道的小功能，当时可能需要1-2名程序员花费三到五天的时间完成。

“码农”现象的兴起

行业内将这些从事重复性、低创造性工作的执行者调侃为”码农”。这个群体面临着明显的职业瓶颈：他们难以处理模糊、复杂的系统性难题，缺乏对业务的深入理解和独立思考，仅仅作为需求的执行终端。他们的工作内容往往等同于将代码从A处复制到B处，遇到问题时更多地依赖加班和”肝”来强行解决。

AI时代：职业分水岭的加速器

降维打击的到来

正如建筑业从人力搬砖发展到机械化施工，编程领域也正在经历类似的变革。AI编程助手的出现，能够更快、更准确地完成那些曾经需要大量人力完成的重复性工作。从公司管理者的角度考虑，支付高额薪水雇佣”码农”来完成AI可以更好完成的工作，已经变得不再经济。

AI技术正在对传统的编程工作模式发起”降维打击”，极大地压缩了基础程序员的生存空间。这种冲击不是渐进的，而是颠覆性的。

重新定义”程序员”与”码农”

在AI时代，我们有必要重新审视”程序员”这一职业的真正内涵：

码农（即将被淘汰的群体）：

  • 局限于代码实现的执行层面
  • 缺乏业务理解和系统设计能力
  • 解决问题的能力单一且表面化
  • 价值主要体现在将需求转换为代码的执行力上

真正的程序员（未来的幸存者与发展者）：

  • 能够解决模糊和复杂的系统性问题
  • 深刻理解业务价值和技术实现的结合点
  • 具备架构设计和技术决策能力
  • 价值体现在判断力、创造力和技术领导力上

AI是敌是友？技术演进的双面性

拉平效应还是放大效应？

有一种观点认为，AI会减少初级程序员与高级程序员在代码开发能力上的差距，从而拉平开发水平。然而，事实可能恰恰相反。

AI的使用，实际上会将高级程序员的优势发挥得更加淋漓尽致。当基础的编码工作被AI自动化后，真正的价值差异就体现在那些AI无法替代的能力上：复杂问题拆解、系统架构设计、业务场景抽象、技术方案选型等高级技能。

丛林法则的新时代

我们正在进入一个技术人才的”丛林时代”，具备优秀问题解决能力的程序员将获得前所未有的技术体验和职场优势。而仍然停留在”码农”思维的执行者，将面临严峻的生存挑战。

在这个新时代，码农，将死。

PIMPL

Posted on 2025-10-04

介绍

Pimpl 的全称是 Pointer to IMPLementation，中文直译就是 “指向实现的指针”，有时也被叫 Cheshire Cat Idiom （像柴郡猫一样，头文件只露出笑容（接口），身体（实现）藏起来）Compilation Firewall（编译防火墙，挡住依赖扩散）

Pimpl 不是语法特性，而是一种设计思想/习惯用法（idiom）。就像 RAII、Singleton 一样，它是 C++ 社区长期演进出来的设计模式。

主要目标是：

  • 隐藏实现细节（information hiding）;

  • 减少编译依赖（降低头文件之间的耦合）;

  • 保持 ABI 稳定（对外接口二进制兼容）;

普通类：.h 文件 = “全身裸露”，别人一眼就看到内部细节。

Pimpl：.h 文件 = “只露个影子（指针）”，别人只知道有个实现，却不知道里面长啥样。

实例

这里只是告诉编译器有一个叫 Impl 的结构体存在，但不说明它里面有些什么成员。

编译器因此知道 Impl 这个名字，但不能直接用它的大小（sizeof(Impl)）或访问里面的字段。

  • 封装实现：头文件只暴露接口，不暴露内部细节；

  • 减少编译依赖：修改 Impl 的实现不会导致依赖 MyClass.h 的代码重新编译；

  • 稳定 ABI：常用于库的接口设计；

内存示意布局

要点：

  • MyClass 对外只有一个指针大小（x64 通常 8 字节）；

  • 大小稳定，ABI 友好；

  • 实际“重量级”成员都藏在 Impl 里，实现细节不泄露。

  • 改 Impl 的字段不会触发大量重编译（依赖减少），编译更快。

和传统的.h的声明方式的对比

  1. 普通类：声明在 .h，实现放 .cpp

特点：

  • .h 文件必须包含所有成员定义（比如 int age, std::string name）；

  • 只不过函数体（实现细节）放在 .cpp；

这样依赖 Animal.h 的代码，会被迫看到 STL 的 <string>，编译时依赖很大。

一旦 Animal 的成员变量变动（比如 std::string 改成 std::vector），所有包含 Animal.h 的代码都得重新编译。

  1. Pimpl 手法（指针实现）

特点：

  • .h 文件 不暴露任何成员变量，只暴露一个指针;

  • .cpp 才包含 等庞大依赖;

Animal.h 变得非常轻量，改动 Impl 时不需要重新编译依赖方，接口 ABI 稳定（sizeof(Animal) 恒定=一个指针大小）。

对比点 普通声明+实现 Pimpl 手法
成员变量 .h 里要写 .h 里看不到
编译依赖 改成员就要全量编译 只改 .cpp 就行
ABI 稳定性 sizeof(Animal) 随成员变化而变 恒定（一个指针大小）
封装性 依赖方能看到类的完整定义 依赖方完全不知道实现细节
