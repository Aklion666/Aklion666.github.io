PIMPL

Posted on 2025-10-04

介绍

Pimpl 的全称是 Pointer to IMPLementation，中文直译就是 “指向实现的指针”，有时也被叫 Cheshire Cat Idiom （像柴郡猫一样，头文件只露出笑容（接口），身体（实现）藏起来）Compilation Firewall（编译防火墙，挡住依赖扩散）

Pimpl 不是语法特性，而是一种设计思想/习惯用法（idiom）。就像 RAII、Singleton 一样，它是 C++ 社区长期演进出来的设计模式。

主要目标是：

  • 隐藏实现细节（information hiding）;

  • 减少编译依赖（降低头文件之间的耦合）;

  • 保持 ABI 稳定（对外接口二进制兼容）;

普通类：.h 文件 = “全身裸露”，别人一眼就看到内部细节。

Pimpl：.h 文件 = “只露个影子（指针）”，别人只知道有个实现，却不知道里面长啥样。

这样做的好处：

  • 封装实现：头文件只暴露接口，不暴露内部细节；

  • 减少编译依赖：修改 Impl 的实现不会导致依赖 MyClass.h 的代码重新编译；

  • 稳定 ABI：常用于库的接口设计；

内存示意布局

要点：

  • MyClass 对外只有一个指针大小（x64 通常 8 字节）；

  • 大小稳定，ABI 友好；

  • 实际“重量级”成员都藏在 Impl 里，实现细节不泄露。

  • 改 Impl 的字段不会触发大量重编译（依赖减少），编译更快。

和传统的.h的声明方式的对比

  1. 普通类：声明在 .h，实现放 .cpp

特点：

  • .h 文件必须包含所有成员定义（比如 int age, std::string name）；

  • 只不过函数体（实现细节）放在 .cpp；

这样依赖 Animal.h 的代码，会被迫看到 STL 的 <string>，编译时依赖很大。

一旦 Animal 的成员变量变动（比如 std::string 改成 std::vector），所有包含 Animal.h 的代码都得重新编译。

  1. Pimpl 手法（指针实现）

特点：

  • .h 文件 不暴露任何成员变量，只暴露一个指针;

  • .cpp 才包含 等庞大依赖;

Animal.h 变得非常轻量，改动 Impl 时不需要重新编译依赖方，接口 ABI 稳定（sizeof(Animal) 恒定=一个指针大小）。

对比点 普通声明+实现 Pimpl 手法
成员变量 .h 里要写 .h 里看不到
编译依赖 改成员就要全量编译 只改 .cpp 就行
ABI 稳定性 sizeof(Animal) 随成员变化而变 恒定（一个指针大小）
封装性 依赖方能看到类的完整定义 依赖方完全不知道实现细节

IPC流量的概念

Posted on 2025-10-03

在App服务商/商业运营商/IPC(网络摄像机）的场景中，我们经常会需要强度流量的概念。主要原因有以下几点：

流量是成本的直接提现

  • 服务器带宽通常是按照**出网流量(egress traffic)**或者带宽峰值来计费的。
  • 公网环境下，云厂商(阿里云、腾讯云、AWS等)都会按流量来计费，比如1GB出网几毛到几块等，看似不多，但是累积下来就是大头。
  • 对运营商而言，流量决定了对外链路的租用费用，骨干网成本。

IPC视频场景流量消耗极大

  • 一路IPC 1080P@15fps H.264编码，码率大概1.5~2Mbps。
  • 如果1000路摄像头同时上传，带宽需求 = 2000 Mbps = 250MB/s。
    • 每天流量：250MB/s x 86400s = 21TB/天。
    • 一个月就是 600 +TB，成本非常高。
  • 所以每增加一路摄像头，都会带来流量线性增长。

流量决定了架构设计

  • 直连（P2P）：流量只在端与端之间走，服务器只做信令，这样流量成本低级。
  • 转发（Relay)：所有流量走App服务器 -> 服务器出网流量爆炸，成本非常高。
  • CDN分发：单源推流，多端观看，利用CDN节点分发，单节点压力小，但CDN出口流量费依然巨大。

影响商业模式

  • ToB项目：客户可能要求本地化部署，就是为了避免云端流量费用。
  • ToC项目：用户端看回放、看直播、云厂商的带宽账单决定盈利模式能否跑通的核心。
  • 很多云存、云回放、视频AI服务都需要额外收费，本质就是流量带来的成本。

流量与体验的矛盾

  • 流量压缩：降低码率 -> 画质下降。
  • 提升画质：提高码率 -> 流量成本升高。
  • 所以经常要在以下方面做平衡：
    • 清晰度
    • 延迟
    • 流程成本

H264的16x16与H265的64x64

Posted on 2025-09-22

宏块的基本概念

宏块块（Macroblock）是H.264编码的基本单位，它通常为16x16像素的亮度(Luma)。
对应的色度（Chroma）部分大小取决于采样格式：

  • 4:2:0 格式（最常见）：每个宏块包含两个 8 × 8 的色度块（U 和 V）。
  • 4:2:2 格式：色度是两个 8 × 16。
  • 4:4:4 格式：色度与亮度一样大，也是 16 × 16。

👉 所以在常见的 YUV 4:2:0 里，一个宏块由：

  • 16 × 16 = 256 个亮度像素
  • 8 × 8 = 64 个 U 色度像素
  • 8 × 8 = 64 个 V 色度像素
    总计 384 个采样点

为什么是16x16

选择 16×16 是编码效率和复杂度之间的折中：

  • 足够大，可以容纳运动估计时的主要内容块。
  • 足够小，可以跟踪物体的局部运动变化。
  • 在运动补偿时，宏块是预测、变换和熵编码的基本单元

宏块内部结构

在 H.264 中，一个 16×16 宏块可以进一步划分：

  • 帧内预测（Intra Prediction）：宏块可分成 16×16、8×8 或 4×4 子块进行预测。
  • 帧间预测（Inter Prediction）：

    宏块可以划分成
  • 16×16
  • 16×8
  • 8×16
  • 8×8（还可以继续划分成 4×8、8×4、4×4）

    这样可以灵活适应不同区域的运动。

宏块与编码工具的关系

每个宏块会进行：

  • 预测（帧内 or 帧间）
  • 残差变换（通常是 4×4 或 8×8 DCT-like 变换）
  • 量化
  • 熵编码（CAVLC/CABAC）

H265的CTU

在H265中，就没有宏块的定义，而是CTU（Coding Tree Unit）。CTU为64x64的块，是编码单元尺寸的升级而带来的压缩效率的提升。想象一下，在过去H264一次性只能压缩16X16的块状的图像，现在H265一次可以压缩64X64的图像。

64x64 CTU的划分方式与16x16的划分方式一样，也是通过宏块的划分。可以继续切分为CU(coding unit)，PU(Prediction Unit)，TU(Transform Unit)，最小可以达到4X4。

为什么可以使用CTU

大块区域预测效率更高

视频里大面积的平坦区域（比如天空、墙壁、背景等），用 64×64 一次编码，比用很多个 16×16 宏块更省比特。减少了 预测模式 + 运动矢量 + 残差信息 的开销。

支持灵活的分割结构

H.264 的宏块划分方式比较有限（16×16 / 8×16 / 16×8 / 8×8）。

H.265 的 CTU 可以分裂成 四叉树结构：
64×64 → 32×32 → 16×16 → 8×8 → 4×4

每层都能单独选择预测方式（帧内/帧间），适配不同区域。

👉 大块区域用大块，小细节用小块，效率远超 H.264 的固定划分。

变换块更灵活

H.264 主要用 4×4 / 8×8 DCT 变换。

H.265 允许从 4×4 到 32×32 的变换单元 (TU)。

平滑区域 → 大 TU（32×32），压缩效率高。

边缘/纹理多的区域 → 小 TU（4×4），细节保留好。

整体压缩率提升

由于块更大、分割更灵活，H.265 在相同主观质量下可以比 H.264 节省 30–50% 的码率。
这是 HEVC 的核心卖点。

成本

当然，H.265 也不是没有代价：

更大的块 + 四叉树划分 → 编码复杂度提高很多。
解码也比 H.264 更耗算力。
所以 H.265 对硬件加速和芯片支持依赖更强（手机、机顶盒、摄像头必须支持硬解码，否则发热/耗电大）。

这张图直观展示了差别：

左边：H.264 宏块 (16×16)，固定划分，通常再细分成 4×4 或 8×8 小块。

右边：H.265 CTU (64×64)，支持递归划分成 32×32 → 16×16 → 8×8 → 4×4，更灵活。

👉 H.265 之所以更有优势，就是因为大块区域能用大 CTU 一次处理，小细节还能继续细分，从而同时提升了压缩率和灵活性。

C++线程中detach与join的选择

Posted on 2025-09-05 Edited on 2025-09-29

前言

在进行多线程编程的时候，如何管理线程的生命周期一直是一件很头疼的事情。如果生命周期管理不好，轻则程序进行过程中马上崩溃，重则在长期运行后因为资源释放失败而造成了虚拟内存碎片的上涨，很难以排查。
在我的实际工作中，有两种方式来处理这个问题：

  1. join来等待thread执行结束；
  2. detach让线程自己管理自己的生命周期，外部不做处理。

join 孤独的等待者

detach()无情的渣男

detach的陷阱

很糟糕的一个情况，需要牢记一个特性，当启动了detach后，所有的事物都是独立的，这个时候如果还有对于外部有依赖，那可不是一个合格的渣男噢！会造成程序错误的。

引入RAII的思想

总结

  • 使用 join()：当你需要等待线程完成并获得其结果时
  • 使用 detach()：当你想让线程在后台独立运行，且确保它不会访问已销毁的对象时
  • 永远不要让 std::thread 对象在没有调用 join() 或 detach() 的情况下被销毁
  • 优先考虑 join()，因为它更安全且更容易管理资源
特性 join detach
阻塞 阻塞调用线程，直到目标线程完成后 立即返回
线程所有权 保持所有权，可以随时查看 放弃所有权
资源管理 自动清理线程资源 线程自己清理资源
安全性 安全，确保线程完成 需要确保线程不会访问已经销毁的对象

直播中画面偶发的跳帧问题排查手段

Posted on 2025-09-03

场景

在一次拉取摄像头音视频数据进行播放的时候出现了画面跳帧、高延迟、卡顿的现象，不满足于产品需求，是一个无法忍受的功能场景，故需要进行问题定位后进行排查。

初步判断

大致可以分为两个部分进行问题排查，首先是设备端推流数据是否有问题，其次是媒体服务器转码的数据是否有问题。由于这里的摄像头数据的功能实现代码是完全无感的，所以只能先从媒体服务器这边进行排查。

手段一：视频保存

将推流的数据保存为视频文件，进行本地播放进行查看，看看获取到的数据是否存在问题。

手段二：原始数据保存

使用网络抓包的手段，但由于视频数据较大，且需要仔细的按帧切割所以不采用常规的一些抓包手段。这里可以选择在服务端打开对应的端口进行端口监听，将保存下来的数据原封不动的保存下来（bin)然后按照私有协议进行payload的切割，这里可以查看：

  1. seq与timestamp是否存在异常；
  2. 按帧保存Jpg图片，查看图片是否正常，跳帧数据是否确实丢失；
  3. 结合 1 和 2 进行更细节的排查；

按照上面三个步骤排查后，可以准确定位是否为服务端的问题。如果服务端有问题，按照这三个部分大致也能定位到了。
